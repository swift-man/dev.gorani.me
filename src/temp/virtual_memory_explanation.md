# Virtual Memory란?

Virtual Memory(가상 메모리)는 **프로세스가 실제 물리 메모리(RAM)보다 더 크고, 연속적이며, 독립적인 메모리 공간을 가진 것처럼 보이게 해주는 운영체제 + 하드웨어(MMU) 메커니즘**입니다.

쉽게 말하면:

- 프로그램은 `0x1000`, `0x2000` 같은 **가상 주소(Virtual Address)** 를 사용합니다.
- CPU/MMU는 그 주소를 실제 RAM의 **물리 주소(Physical Address)** 로 변환합니다.
- 이 변환 정보는 운영체제가 관리하는 **페이지 테이블(Page Table)** 에 들어 있습니다.

즉, 프로그램은 실제 RAM 배치를 직접 모르고, **자기만의 주소 공간** 안에서 동작합니다.

---

## 왜 필요한가?

가상 메모리가 없으면 다음 문제가 큽니다.

### 1. 프로세스 격리
프로세스 A가 프로세스 B의 메모리를 직접 건드리면 큰 문제가 됩니다.
가상 메모리는 각 프로세스마다 **독립된 주소 공간**을 제공해서 서로 침범하지 못하게 합니다.

### 2. 메모리 보호
커널 영역, 코드 영역, 읽기 전용 데이터 영역 등에 접근 권한을 다르게 줄 수 있습니다.
예를 들어:

- 어떤 페이지는 읽기만 가능
- 어떤 페이지는 실행 가능하지만 쓰기 불가
- 어떤 페이지는 커널만 접근 가능

이런 식의 보호가 가능합니다.

### 3. 연속적인 메모리처럼 보이게 함
실제 RAM에서는 데이터가 듬성듬성 흩어져 있어도,
프로그램 입장에서는 마치 연속된 큰 배열처럼 사용할 수 있습니다.

### 4. RAM보다 큰 메모리 공간을 다루기 쉬움
필요한 페이지만 RAM에 올리고,
당장 필요 없는 페이지는 디스크(스왑/페이지 파일) 쪽으로 내릴 수 있습니다.
이 덕분에 실제 RAM보다 큰 작업도 어느 정도 처리할 수 있습니다.

---

## 핵심 개념

## 1. Virtual Address vs Physical Address

프로그램이 보는 주소:

```text
Virtual Address
```

실제 RAM 위치:

```text
Physical Address
```

이 둘은 보통 다릅니다.
운영체제와 MMU가 중간에서 매핑합니다.

---

## 2. Page
가상 메모리는 보통 **페이지(Page)** 단위로 관리됩니다.
예: 4KB, 16KB 같은 고정 크기 블록.

- 가상 주소 공간도 페이지 단위로 나뉨
- 물리 메모리도 프레임(Frame) 단위로 나뉨
- 가상 페이지 하나가 물리 프레임 하나에 매핑됨

---

## 3. Page Table
페이지 테이블은 이런 정보를 담습니다.

- 이 가상 페이지가 어느 물리 프레임에 대응되는지
- 현재 RAM에 올라와 있는지
- 읽기/쓰기/실행 권한이 무엇인지
- 최근 접근 여부, 수정 여부 등

---

## 4. MMU
MMU(Memory Management Unit)는 CPU가 메모리에 접근할 때
가상 주소를 물리 주소로 바꿔주는 하드웨어입니다.

실행 흐름은 대략 이렇습니다.

1. 프로그램이 가상 주소 접근
2. MMU가 페이지 테이블 확인
3. 매핑이 있으면 물리 주소로 변환
4. 매핑이 없거나 권한이 없으면 예외 발생

---

## 5. Page Fault
접근한 가상 페이지가 현재 RAM에 없으면 **페이지 폴트(Page Fault)** 가 발생합니다.

운영체제가 개입해서:

- 해당 페이지를 디스크에서 RAM으로 올리거나
- 잘못된 접근이면 프로세스를 종료하거나
- 새 페이지를 할당합니다.

페이지 폴트 자체는 반드시 오류라는 뜻은 아닙니다.
**정상적인 demand paging 과정**일 수도 있습니다.

---

## 동작을 비유하면

가상 메모리는 호텔 프런트와 객실 배정표에 가깝습니다.

- 손님(프로세스)은 자기 방 번호(가상 주소)만 압니다.
- 실제 어느 구역의 어느 방(물리 주소)에 배정됐는지는 프런트(운영체제/MMU)가 관리합니다.
- 어떤 방은 지금 준비 중이라 잠깐 기다려야 할 수 있는데, 이게 page fault에 가깝습니다.

즉, 사용자는 단순한 번호 체계만 보고,
실제 배치는 시스템이 뒤에서 처리합니다.

---

## 개발자가 알아야 할 실무 포인트

### 1. 가상 메모리 = 스왑 이 아니다
많이 헷갈리는데, 가상 메모리의 핵심은 **주소 추상화와 보호**입니다.
스왑은 그 위에서 동작할 수 있는 기능 중 하나일 뿐입니다.

### 2. 포인터는 가상 주소를 가리킨다
애플리케이션 코드에서 다루는 포인터는 보통 물리 주소가 아니라 **프로세스의 가상 주소 공간 안의 주소**입니다.

### 3. 메모리 사용량과 실제 RAM 사용량은 다를 수 있다
프로세스가 큰 주소 공간을 예약했다고 해서,
그게 곧바로 같은 양의 실제 RAM을 차지하는 것은 아닙니다.

예를 들어:

- 큰 메모리를 `mmap` 했지만 아직 접근 안 함
- copy-on-write로 공유 중임
- 일부 페이지만 실제로 commit 됨

이런 경우가 있습니다.

### 4. 성능 문제는 페이지 단위로 터진다
메모리 접근 패턴이 나쁘면:

- TLB miss 증가
- page fault 증가
- 캐시 locality 저하

로 이어져 성능이 급격히 나빠질 수 있습니다.
대용량 배열, 이미지 처리, DB, 머신러닝, 게임 엔진 같은 분야에서 중요합니다.

---

## 예시로 이해하기

어떤 프로세스가 아래처럼 8KB를 사용한다고 가정해봅시다.
페이지 크기가 4KB라면 2페이지가 필요합니다.

```text
Virtual Page 0 -> Physical Frame 12
Virtual Page 1 -> Physical Frame 91
```

프로세스 입장에서는 메모리가 연속적입니다.
하지만 실제 RAM에서는 12번 프레임과 91번 프레임처럼 떨어져 있을 수 있습니다.
그래도 페이지 테이블이 매핑해 주므로 문제없이 동작합니다.

---

## 한 줄 정의

**Virtual Memory는 각 프로세스에 독립적이고 연속적인 주소 공간을 제공하고, 그 주소를 실제 물리 메모리에 매핑해 메모리 보호, 격리, 효율적 사용을 가능하게 하는 시스템입니다.**

---

## 면접에서 이렇게 말하면 좋음

> 가상 메모리는 프로세스가 실제 RAM과 분리된 독립적인 주소 공간을 사용하게 하는 메커니즘입니다. CPU는 가상 주소를 사용하고, MMU와 페이지 테이블이 이를 물리 주소로 변환합니다. 이를 통해 프로세스 격리, 메모리 보호, demand paging, 효율적인 메모리 관리가 가능해집니다.
