# Garbage Collection이란?

Garbage Collection(GC)은 **프로그램이 더 이상 참조하지 않는 메모리(객체)를 자동으로 찾아서 해제하는 메모리 관리 방식**입니다.

개발자가 `free`, `delete`, `dealloc` 같은 해제를 직접 하지 않아도, 런타임이 “이 객체는 이제 살아 있을 필요가 없다”고 판단하면 정리해 줍니다.

---

## 왜 필요한가?

수동 메모리 관리에서는 보통 이런 문제가 생깁니다.

- **메모리 누수(Memory Leak)**: 해제해야 할 객체를 계속 남겨 둠
- **이중 해제(Double Free)**: 이미 해제한 메모리를 또 해제함
- **댕글링 포인터(Dangling Pointer)**: 해제된 메모리를 계속 참조함

GC는 이런 문제 중 특히 **“언제 해제해야 하는가”** 를 런타임이 대신 판단해 준다는 점에서 생산성이 큽니다.

---

## 핵심 개념

GC의 핵심은 단순합니다.

> **도달 가능한(reachable) 객체는 살리고, 더 이상 도달할 수 없는(unreachable) 객체는 수거한다.**

예를 들어:

```text
root -> A -> B
       \
        -> C
```

이때 루트(root)에서 따라가서 접근 가능한 `A`, `B`, `C`는 살아 있는 객체입니다.
반대로 어떤 객체 `D`가 어디에서도 참조되지 않으면, GC는 그 객체를 **garbage** 로 보고 회수합니다.

여기서 root는 보통 이런 것들입니다.

- 현재 실행 중인 스레드의 스택 변수
- 전역 변수 / static 변수
- CPU 레지스터에 잡힌 참조
- 런타임이 관리하는 핸들

---

## 동작 방식

GC 구현은 여러 가지가 있지만, 대표적으로 **Tracing GC** 계열이 많습니다.

### 1. Mark
루트에서 시작해서 참조 가능한 객체를 전부 따라가며 **“살아 있음” 표시**를 합니다.

### 2. Sweep
힙 전체를 훑으면서 표시되지 않은 객체를 **정리**합니다.

이걸 흔히 **Mark-and-Sweep** 이라고 부릅니다.

개념적으로는 이런 흐름입니다.

```text
1) Root 집합 확인
2) Root에서 도달 가능한 객체들 mark
3) mark되지 않은 객체 sweep
```

---

## 대표적인 GC 기법

### 1. Mark and Sweep
가장 기본적인 형태입니다.

장점:
- 개념이 직관적
- 순환 참조도 처리 가능

단점:
- sweep 후 메모리가 듬성듬성 비어 **단편화(fragmentation)** 가 생길 수 있음
- GC 시점에 멈춤 시간이 생길 수 있음

### 2. Mark and Compact
정리 후 살아 있는 객체를 한쪽으로 모아서 **압축(compaction)** 합니다.

장점:
- 단편화 완화
- 큰 연속 메모리 할당에 유리

단점:
- 객체를 이동시키므로 참조 업데이트 비용이 듦

### 3. Copying GC
메모리 공간을 두 영역으로 나눠서, 살아 있는 객체만 다른 쪽으로 복사합니다.

장점:
- 할당이 빠름
- 단편화가 적음

단점:
- 메모리 여유 공간이 더 필요함

### 4. Generational GC
“대부분의 객체는 빨리 죽는다”는 경험 법칙을 이용합니다.

- **Young Generation**: 금방 생성되고 금방 사라지는 객체
- **Old Generation**: 오래 살아남은 객체

젊은 영역은 자주, 빠르게 수집하고
오래된 영역은 덜 자주 수집합니다.

현대 JVM, .NET 런타임 등은 이런 세대 기반 최적화를 적극적으로 씁니다.

---

## Stop-the-World란?

GC를 설명할 때 자주 나오는 말이 **Stop-the-World(STW)** 입니다.

이건 GC가 메모리 그래프를 안전하게 분석하거나 정리하는 동안,
**애플리케이션 실행을 잠시 멈추는 것**을 뜻합니다.

문제는 STW 시간이 길어지면:

- UI 끊김
- 서버 응답 지연
- 프레임 드랍

같은 성능 문제가 생긴다는 점입니다.

그래서 현대 GC는 다음을 개선하려고 합니다.

- **Incremental GC**: 한 번에 다 하지 않고 조금씩 나눠서 수행
- **Concurrent GC**: 애플리케이션과 최대한 동시에 수행
- **Parallel GC**: 여러 스레드로 GC 작업 수행

즉, 좋은 GC의 핵심은 단순히 “잘 치우는 것”이 아니라
**지연(latency), 처리량(throughput), 메모리 사용량** 사이 균형을 맞추는 것입니다.

---

## Reference Counting과의 차이

많이 헷갈리는 부분입니다.

**Reference Counting(참조 카운팅)** 은 객체를 참조하는 수를 세다가,
카운트가 0이 되면 즉시 해제합니다.

장점:
- 해제 시점이 비교적 예측 가능
- STW가 거의 없음

단점:
- **순환 참조(cycle)** 를 스스로 처리하지 못함

예:

```text
A -> B
B -> A
```

외부에서 아무도 `A`, `B`를 참조하지 않아도,
서로가 서로를 잡고 있어서 카운트가 0이 되지 않을 수 있습니다.

반면 **Tracing GC** 는 루트에서 도달 가능한지만 보기 때문에,
이런 순환 구조라도 외부에서 닿지 않으면 수거할 수 있습니다.

## Swift의 ARC는 GC인가?

아닙니다.

Swift/Objective-C의 **ARC(Automatic Reference Counting)** 는 이름은 자동이지만,
엄밀히 말하면 **GC가 아니라 참조 카운팅 기반 자동 메모리 관리**입니다.

즉:

- Swift ARC = Reference Counting
- Java / Go / C# 런타임의 GC = Tracing GC 중심

그래서 Swift에서는 `weak`, `unowned` 를 사용해 **순환 참조를 직접 끊어야** 합니다.
이건 GC 언어와 중요한 차이입니다.

---

## GC가 있다고 메모리 문제가 사라지나?

그렇지는 않습니다.

GC가 있어도 여전히 이런 문제는 생깁니다.

### 1. Logical Memory Leak
객체가 실제로는 필요 없는데, 코드상 참조가 남아 있어서 계속 살아 있는 경우입니다.

예:
- 전역 캐시에 무한히 쌓아 둠
- 이벤트 리스너 해제 안 함
- static collection에 계속 보관함

GC 입장에서는 “참조되고 있으니 살아 있는 객체”이므로 해제하지 않습니다.
즉, **GC는 필요 없는 객체를 추측해서 없애는 도구가 아니라, 도달 불가능한 객체를 정리하는 도구**입니다.

### 2. GC Pause
객체가 너무 많이 생성되거나 힙이 커지면 GC 비용이 커질 수 있습니다.

### 3. Allocation Pressure
짧은 생명주기의 객체를 과도하게 만들면 GC가 매우 자주 돌 수 있습니다.

서버나 앱 성능 최적화에서는 종종 “GC를 없애는 것”보다
**불필요한 객체 생성을 줄이는 것**이 더 중요합니다.

---

## 개발자 관점에서 중요한 포인트

### 1. 메모리 해제를 직접 안 해도, 객체 생명주기는 여전히 설계해야 함
GC 언어에서도 객체를 오래 잡고 있으면 메모리가 계속 늘어납니다.

### 2. 큰 컬렉션, 캐시, 콜백, 리스너는 항상 의심해야 함
참조가 남는 구조는 GC 환경에서도 누수처럼 보입니다.

### 3. 성능은 “객체 수”와 “할당 패턴”의 영향을 크게 받음
짧은 객체를 폭발적으로 만들면 young GC 빈도가 올라갑니다.

### 4. 지연 민감한 시스템에서는 GC 특성을 반드시 알아야 함
게임, 실시간 시스템, 저지연 서버는 GC pause 특성이 중요합니다.

---

## 한 줄 정리

**Garbage Collection은 더 이상 도달할 수 없는 객체를 런타임이 자동으로 찾아 회수하는 메모리 관리 방식**입니다.

추가로 아주 짧게 구분하면:

- **GC**: 도달 가능성 기반으로 자동 회수
- **ARC/참조 카운팅**: 참조 수 기반으로 자동 해제

즉, 둘 다 “자동 메모리 관리”지만, **동작 원리는 다릅니다.**
